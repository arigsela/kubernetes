# Kagent IDP v1.8 — Backstage Kubernetes Tab for Agents

**Date:** 2026-05-19
**Status:** Draft
**Owner:** group:platform-engineering

## Goal

Make the live Kubernetes resources for each kagent agent (Pod,
Deployment, ReplicaSet, Service, ServiceAccount, plus the Agent CRD
itself) visible in Backstage. Today the kagent-agent entity falls
through to a minimal 2-tab layout (Overview + Docs) with no Kubernetes
visibility, even though the entity has the right annotations for the
K8s plugin to work.

## Non-goals

- ModelConfig / RemoteMCPServer CRDs as Backstage entities. Those are
  shared cluster-wide, not per-agent — they don't fit the per-agent
  entity model. Defer to a future iteration if needed.
- A custom "Delegate graph" visualization tab. v1.7 already shows the
  delegate list in the About card; SVG-rendered graph is v1.9+ scope.
- A custom Backstage frontend plugin. The standard Kubernetes plugin
  with `customResources` config is sufficient.
- Multi-cluster support. Hardcoded to `homelab` cluster; multi-cluster
  pattern is future work.

### Why the label changes are needed

Today the entity's `backstage.io/kubernetes-label-selector` is auto-generated by TeraSky from the **Deployment**'s labels (which kagent's controller sets when spawning the Deployment from the Agent CRD):

```
app=kagent,app.kubernetes.io/managed-by=kagent,app.kubernetes.io/name=homelab-knowledge
```

The K8s plugin uses this selector to fetch any K8s object whose labels match all three. The Pod, Deployment, ReplicaSet, Service, and ServiceAccount that kagent spawns have all three labels (the controller sets them). The Agent CRD itself (the one we write into git) has different labels (`part-of: kagent`, `managed-by: backstage-scaffolder`) and does NOT match. So Custom Resources panel in the K8s tab would be empty even with `customResources` config.

Adding `app=kagent`, `app.kubernetes.io/managed-by=kagent`, and `app.kubernetes.io/name=<name>` to the Agent CRD's labels makes it match the selector. The Agent CRD will then be fetched alongside the workload resources.

### Why the decommission check migration

The existing decommission action's safety check uses
`metadata.labels["app.kubernetes.io/managed-by"] === "backstage-scaffolder"`
to verify an agent is IDP-managed before opening a teardown PR. If we
change that label value to `kagent` (required for the K8s selector
match), the safety check would no longer fire and the action could
accidentally delete hand-crafted agents.

Migrating the safety check to a new dedicated label
(`arigsela.com/idp-managed: "true"`) decouples the "IDP-managed?"
signal from the "what tool manages this in the cluster?" signal. The
new label is semantically clearer:

- `app.kubernetes.io/managed-by: kagent` = the **runtime** manager (kagent's controller spawns Deployments from this CRD)
- `arigsela.com/idp-managed: "true"` = the **provenance** signal (this CRD was created via the Backstage IDP)

The decommission action only reads the YAML body from GitHub via
Octokit — it doesn't observe live cluster state — so the migration is
purely a YAML-file + TypeScript change with no propagation timing
concerns.

## The new `kagentAgentPage` layout

```typescript
const kagentAgentPage = (
  <EntityLayout>
    <EntityLayout.Route path="/" title="Overview">
      {overviewContent}
    </EntityLayout.Route>

    <EntityLayout.Route
      path="/kubernetes"
      title="Kubernetes"
      if={isKubernetesAvailable}
    >
      <EntityKubernetesContent />
    </EntityLayout.Route>

    <EntityLayout.Route path="/docs" title="Docs">
      {techdocsContent}
    </EntityLayout.Route>
  </EntityLayout>
);
```

Routing (one new case in the existing `EntitySwitch`):

```diff
 const componentPage = (
   <EntitySwitch>
     <EntitySwitch.Case if={isComponentType('service')}>
       {serviceEntityPage}
     </EntitySwitch.Case>

     <EntitySwitch.Case if={isComponentType('website')}>
       {websiteEntityPage}
     </EntitySwitch.Case>

+    <EntitySwitch.Case if={isComponentType('kagent-agent')}>
+      {kagentAgentPage}
+    </EntitySwitch.Case>

     <EntitySwitch.Case>{defaultEntityPage}</EntitySwitch.Case>
   </EntitySwitch>
 );
```

### What's intentionally NOT in the new layout

- **CI/CD tab** — kagent agents don't have a CI pipeline; the generic `cicdContent` would render "No CI/CD available for this entity".
- **API tab** — agents don't provide/consume OpenAPI or GraphQL APIs.
- **Dependencies tab** — no useful component dependencies to render.
- **Crossplane tab** — `isCrossplaneAvailable` predicate fails for non-XRs.

Each of those would render either an empty state or a misleading "no data" panel, so excluding them is cleaner. The Docs tab is included even though it's currently empty (no `backstage.io/techdocs-ref`) to leave the door open for per-agent TechDocs without another EntityPage change.

## The `kubernetes.customResources` config

Both files get the same block.

**`app-config.yaml`** — add inside the existing `kubernetes:` block, after `clusterLocatorMethods`:

```yaml
kubernetes:
  serviceLocatorMethod:
    type: 'multiTenant'
  clusterLocatorMethods:
    - type: 'config'
      clusters:
        - url: ${K8S_CLUSTER_URL}
          name: homelab
          authProvider: 'serviceAccount'
          skipTLSVerify: true
          serviceAccountToken: ${K8S_SERVICE_ACCOUNT_TOKEN}
  customResources:
    - group: 'kagent.dev'
      apiVersion: 'v1alpha2'
      plural: 'agents'
```

**`app-config.production.yaml`** — add a minimal `kubernetes:` block (today has none):

```yaml
kubernetes:
  customResources:
    - group: 'kagent.dev'
      apiVersion: 'v1alpha2'
      plural: 'agents'
```

The v1.6 finding documented that production replaces arrays. Adding `customResources` to both files keeps the entry present regardless of which config layer is loaded.

## Rollback plan

If the migration breaks something:

1. Revert the EntityPage.tsx + app-config changes (Backstage rollback to prior image)
2. Revert the scaffolder template change
3. Revert the decommission action's label name back to `app.kubernetes.io/managed-by=backstage-scaffolder`
4. Revert `homelab-knowledge.yaml` to its prior label set

Worst case: 4 small revert PRs. Decommission action would refuse all agents until the label matches the action's check — annoying but safe.

## Known limitations

1. **Single cluster hardcoded.** The K8s plugin's cluster locator is configured with just `homelab`. Multi-cluster deployments would need to read the cluster name from the entity (e.g. `backstage.io/kubernetes-cluster` annotation).
2. **ModelConfigs not shown.** They're shared cluster-wide and don't match per-agent label selectors. Future work could create a separate "kagent platform" page or a ModelConfig entity.
3. **No live pod-log streaming on this tab.** The K8s plugin's log viewer works but requires a click into the pod; no continuous tail. (Standard Backstage behavior.)

## Open follow-ups (not v1.8 scope)

- **ModelConfig visibility** — separate entity or shared "kagent platform" page.
- **Health alerting integration** — surface Prometheus/Grafana alerts for the agent's Deployment on the entity page.
- **Multi-cluster cluster locator** — if/when the homelab grows beyond one cluster.
